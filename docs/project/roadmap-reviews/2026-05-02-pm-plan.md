### Stage 3: MOSS-NANO Productization Track

#### MOSS-NANO-13a — Real Sidecar Adapter

- **What:** Replace the stub `main/moss-nano-sidecar.js` (currently returns `sidecar-adapter-not-configured` for every method) with a real Python subprocess adapter that satisfies the existing engine contract: spawn the Nano runtime as a subprocess, frame stdin/stdout messages, track PID, reap zombies, propagate lifecycle generation across restarts. Wire to a productized version of `scripts/moss_nano_resident_probe.py` so the integrated `nanoSynthesize` IPC call actually exercises the same Python runtime that produced the 6F bounded-soak evidence.
- **Why:** Audit findings NANO13-M1 and integrated-app reality. Without a real adapter, the integrated app cannot synthesize Nano audio in any reading mode. The 6F runtime evidence (1800s soak, p95 first-decoded 280ms, zero stale output) was produced by `moss_nano_resident_probe.py` running standalone — not by the integrated `moss-nano-engine.js` lifecycle. MOSS-NANO-13e cannot capture live evidence until this chain is wired end-to-end.
- **Prerequisites:** EINK-6B and GOALS-6B do not block; runs in parallel with them via lane separation (Lane D — Platform/Main Process).
- **Done when:** Defined in the eager skeleton in Phase D.
- **Effort:** L (~8). Real systems work. Plausible expansion if subprocess lifecycle quirks surface.
- **Roster:** CLI/Zeus orchestration → Athena (cross-system implementation).
- **Source:** Audit response §10 item 3, audit findings NANO13-M1.

#### MOSS-NANO-13b — Engine Hardening + Strategy Invariants

- **What:** Three coupled fixes. (1) Enforce `synthesizeTimeoutMs` and `commandTimeoutMs` in `main/moss-nano-engine.js` via `Promise.race` against `setTimeout`-rejected sentinel; add contract test. (2) Fix `setContinuityScope` in `src/hooks/narration/mossNanoStrategy.ts:412-415` to bump `generationId` (not just `prefetchGenerationId`); add adversarial test for `speakChunk` mid-flight + scope change → late synthesize result is NOT scheduled. (3) Upgrade `hashText` (`src/hooks/narration/mossNanoStrategy.ts:87-93`) — either include `text.length` in the cache key or upgrade to a 64-bit non-cryptographic hash with low collision rate over typical chunk sizes.
- **Why:** Audit findings NANO13-M2, NANO13-M4, NANO13-MOD1. Decorative timeouts hang the synthesize promise indefinitely on a hung adapter. `setContinuityScope` leaves cross-section staleness as a renderer-discipline property, not a strategy invariant. 32-bit hash is a soundness softening.
- **Prerequisites:** MOSS-NANO-13a (so timeouts have a real adapter to bound; though some pieces can land before 13a).
- **Done when:** Defined in the eager skeleton in Phase D.
- **Effort:** M (~3). Tightly bounded fixes with explicit edit sites.
- **Roster:** CLI/Zeus orchestration → Hephaestus (electron-scope for engine + renderer-scope for strategy).
- **Source:** Audit response §10 items 4, 5, MOD1.

#### MOSS-NANO-13c — Live Evidence Schema + Producer + Gate Validation

- **What:** Three coupled deliverables. (1) Redefine the `liveEvidence.modes[mode]` schema as machine-produced: each entry must include `runArtifactPath`, `traceEventCount`, `nanoSegmentLatencyMs.{p50,p95,min,max}`, `nanoCache.{hits,misses,hitRate}`, `nanoPrefetch.{ready,stale,cancelled}`, `recycleObservations`, `recordedAt`, `appCommit`, `evidenceProducerVersion`, plus a schema version field. (2) Build the evidence producer — a new tool (likely `scripts/moss_nano_live_capture.mjs` or a new mode in `tts_eval_runner.mjs`) that exercises the integrated app in `--nano-live-capture` mode, records real `nano-segment` events from `mossNanoStrategy.onSegmentTrace`, and emits the sealed live-evidence JSON. (3) Update `evaluateMossNanoLiveEvidenceGate` at `scripts/tts_eval_runner.mjs:100-160` to validate provenance: assert run artifact exists on disk, `traceEventCount` matches `summary.json`, latency and cache numbers are within the artifact's bounds, no key is `true` without a non-null quantitative basis. Update or remove the existing matrix-simulator passing-evidence test (`tests/ttsEvalMatrixRunner.test.ts:709-767`) since the gate now rejects hand-authored boolean JSON.
- **Why:** Audit findings NANO13-C1 (gate is structurally unverifiable). The gate currently passes `NANO_RECOMMENDED_OPT_IN` for any hand-edited JSON of all-true booleans; this is a worse posture than today's `NANO_EXPERIMENTAL_ONLY`. Audit response §10 items 1 and 2.
- **Prerequisites:** MOSS-NANO-13a (need real adapter so producer can record real events) + MOSS-NANO-13b (need scope/timeout/cache invariants holding so recorded events are meaningful).
- **Done when:** Defined in the eager skeleton in Phase D.
- **Effort:** L (~8). New tool category + schema design + gate refactor + test rewrite.
- **Roster:** CLI/Zeus orchestration → Athena (architectural cross-system) + Hippocrates (gate test rewrites).
- **Source:** Audit response §10 items 1–2, audit findings NANO13-C1, NANO13-MOD2.

#### MOSS-NANO-13d — Audit Memo Reframing & Default-Engine Posture (stub — Stage 3 close)

- **What:** Update `docs/audit/2026-05-02-moss-nano-productization-third-party-audit/AUDIT_MEMO.md` and any orientation docs to name Qwen as the active default and Kokoro as the deprecated legacy fallback. Update the matrix manifest's `nanoGate.kokoroAvailable` field to either `defaultAvailable` or pair with `qwenAvailable` so the gate tracks the actual default. Update settings UI fallback narrative ("Nano failed; switch back to Qwen (or Kokoro)") instead of "Nano failed; Kokoro is your fallback."
- **Why:** Audit finding NANO13-M3. The audit memo materially misframes the engine landscape — the actual default per `ROADMAP.md`, `TTSSettings.tsx`, and `tests/ttsSettingsMossNano.test.tsx:136` is Qwen.
- **Prerequisites:** MOSS-NANO-13c (so the gate is already trustworthy when the rename lands).
- **Done when:** TBD at Stage 3 close.
- **Effort:** S (~1). Documentation + manifest field rename + UI string update.
- **Roster:** CLI/Hermes (mechanical) plus Cowork (memo rewrite).
- **Source:** Audit response §10 item 6.

#### MOSS-NANO-13e — Live Capture Run + Decision (stub — Stage 3 close)

- **What:** Execute the evidence producer against a real selected-Nano session in all four reader modes (Page, Focus, Flow, Narrate). Capture live evidence including recycle-UX observations (worst-case mid-utterance recycle gap, total recycle count during session). Feed the resulting evidence JSON through `evaluateMossNanoLiveEvidenceGate`. Record the decision (`PAUSE_NANO_PRODUCTIZATION`, `NANO_EXPERIMENTAL_ONLY`, or `NANO_RECOMMENDED_OPT_IN`) in `docs/testing/MOSS_DECISION_LOG.md`. Update `ROADMAP.md` finish-line note. If the decision is `NANO_RECOMMENDED_OPT_IN`, update settings UX to surface that recommendation (separate sprint may be needed for the UX wiring depending on scope).
- **Why:** Closes the MOSS-NANO productization gate that was first proposed in MOSS-NANO-13's original AUDIT_MEMO sketch. Audit response §11 final recommendation.
- **Prerequisites:** MOSS-NANO-13a, 13b, 13c, 13d.
- **Done when:** TBD at Stage 3 close, after 13a-d are in.
- **Effort:** M (~3). Run-and-record, plus decision integration.
- **Roster:** Cowork (run execution + decision write-up) • CLI/Zeus (any UX wiring needed if PROMOTE).
- **Source:** Audit response §11.
