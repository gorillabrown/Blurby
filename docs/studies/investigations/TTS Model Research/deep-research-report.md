# Open-source TTS options for Blurby

## Decision summary

On Blurby’s actual product constraints, no model in this set cleanly beats hexgrad’s Kokoro as the total-package choice today. The models most likely to sound more natural than Kokoro in long-form narration are Qwen3-TTS and OpenMOSS’s MOSS-TTS family, but both come with materially heavier runtime, packaging, and Windows support burdens than Blurby’s current local ONNX path. Chatterbox is technically strong and commercially permissive, but its mandatory built-in watermarking and GPU-centric operational profile make it a less clean fit for a desktop reader. MeloTTS is easy to run but does not look like a meaningful narration upgrade over Kokoro. Dia is impressive for dialogue, but its commercial posture is ambiguous in practice and its operational behavior is poorly aligned with single-speaker book narration. citeturn10search0turn13view0turn2view1turn24view3turn6view0turn31view0turn20view0turn20view1turn15view0turn16view0turn18view1turn26view0

The best immediate ship choice for Blurby remains **hexgrad / Kokoro**. The best **prototype candidate** for an attempted upgrade is **OpenMOSS**, specifically MOSS-TTS-Local-Transformer for a higher-quality track and MOSS-TTS-Nano as an operationally interesting side branch. ## Blurby-specific baseline

Blurby is not evaluating these models in a vacuum. It is a Windows-first Electron desktop app, and its current neural narration path is a local Kokoro ONNX worker flow built around sentence-aligned chunking, a rolling audio queue, real word-timestamp support from a patched `kokoro-js`, and a resumable narration pipeline that currently assumes low-latency local inference and deterministic restart behavior. That means a model can sound better in a benchmark and still be a worse Blurby choice if it demands a Python sidecar, large staged assets, GPU-only execution, fragile dependencies, or a separate alignment system. fileciteturn0file0

That baseline matters because Kokoro is already deeply integrated into Blurby’s architecture. Blurby currently uses Kokoro locally through ONNX inference in a worker thread, with a small-asset deployment model and timing-aware chunk orchestration. Replacing that with a heavier autoregressive speech stack would not be a drop-in swap. It would change packaging, startup, caching, re-entry behavior, and likely the UI’s synchronization layer as well. fileciteturn0file0

## Provider findings

**hexgrad** remains the best match to Blurby’s current architecture because Kokoro is small, Apache-licensed, local, and already available through both Python and JavaScript-facing ONNX paths. The ONNX community card documents JavaScript usage, multiple quantizations, and a 512-token context ceiling. The main repo documents Windows installation, explicit speed control, and split-pattern handling for long text. The downside is that Kokoro’s long-form quality ceiling is not best-in-class anymore, and the 510-token practical limit means Blurby still has to do the orchestration work very carefully. That is acceptable for a reader app because Blurby already does that work, but it also means Kokoro is a baseline to beat, not a new discovery. citeturn35view0turn13view0turn22view0turn23search2turn12search1

**OpenMOSS-Team** is the strongest upgrade track if Blurby is willing to prototype a heavier stack. The core family explicitly targets stable long-form speech, pronunciation accuracy, style switching, and output that stays coherent over tens of minutes. The docs also expose things Blurby actually cares about rather than generic TTS demos: fine-grained duration control, a local 1.7B transformer variant, a torch-free `llama.cpp` plus ONNX path, a CPU-only configuration, staged low-memory GPU configs, and a new Nano branch that is explicitly CPU-friendly and long-text capable. The problem is that this is not a small “swap the model” integration. Even the torched-free path still involves multiple assets, model conversion assumptions, and in the current repo a compiled bridge. That makes it the best **prototype** target, but not a low-risk ship path. citeturn27view3turn27view4turn27view0turn31view0turn27view5turn39search0turn39search1

**Qwen** comes from the Qwen team at entity\["company","Alibaba Cloud","cloud computing company"] and has some of the strongest long-speech evidence in this set. The official repository and model cards document 10-language support, 3-second rapid voice cloning, controllable custom speakers, generation kwargs such as `max\_new\_tokens`, and long-speech benchmark results that are unusually strong for content consistency. For pure narration quality, Qwen may be the most credible “sounds better than Kokoro” candidate here. The reason it does not win for Blurby is operational, not acoustic: even the 0.6B base is large on disk, the official docs are GPU-first, FlashAttention is recommended to lower memory use, Windows usage is plausible but not first-party-simple, and production-like runs still seem to require careful `max\_new\_tokens` and segmentation discipline. For a Windows Electron reader, that is a serious tax. citeturn40search1turn40search7turn24view0turn24view1turn24view3turn24view4turn24view5turn30search2turn28search17

**ResembleAI** is the best pure watchlist entry if Blurby later decides that a Python sidecar and GPU-first runtime are acceptable. Chatterbox’s model card is attractive for narration, not just for agents: it claims strong zero-shot English quality, multilingual support, alignment-informed inference, pacing knobs through `cfg\_weight` and `exaggeration`, and a straightforward `pip install` entry. The problem is that every generated file includes Resemble AI’s PerTh watermark. The watermark is documented as imperceptible, so it does not violate the literal “no audible disclaimer” gate, but it is still a permanent output policy artifact that Blurby would need to be comfortable shipping into exported audiobook files. Operationally, the official examples are CUDA-first, the repo was tested on Python 3.11 on Debian, and public issues show CPU-only and Windows friction. That makes it more suitable for experimentation than adoption. citeturn18view1turn26view0turn26view6turn32view0turn25search0turn25search4

**MeloTTS** is published by entity\["company","MyShell.ai","ai company"] and is still one of the cleanest “small local multilingual TTS” projects in open source. It is MIT-licensed, explicitly says CPU is sufficient for real-time inference, and exposes straightforward speaker and speed controls. But for Blurby the question is not whether MeloTTS works. The question is whether it materially improves the reading experience over Kokoro while keeping local deployment cheap. The evidence here says no. It is older, more conventional, and operationally awkward on Windows because the project explicitly suggests Docker to avoid compatibility issues. That combination makes it a poor use of Blurby engineering time. citeturn20view0turn20view1turn20view2turn19search8

**nari-labs** is the easiest reject. Dia is technically interesting, especially for two-speaker expressive dialogue, and the repo shows impressive speed on a 4090-class GPU. But the same docs also say the model was not fine-tuned on a specific voice, so outputs vary run to run unless you manage prompts or seeds; long inputs over roughly 20 seconds become unnaturally fast; the project had only been tested on GPUs at launch; CPU support was still “to be added”; and the model card uses Apache-2.0 while also describing the project as intended for research and educational use with additional forbidden uses. For a single-speaker EPUB-to-audio workflow, that is the wrong shape of model and the wrong shape of risk. citeturn15view0turn16view0turn32view2turn32view4turn32view5

## Final recommendation

If Blurby’s question is, “Which provider is best for us right now?”, the answer is **hexgrad**. Kokoro is still the best total fit because Blurby already depends on exactly the things Kokoro is good at: compact local inference, Windows viability, low packaging burden, deterministic chunk restarts, and timing-friendly integration. It is not the best-sounding system in the list, but it is the best **Blurby** system in the list. fileciteturn0file0 citeturn10search0turn13view0turn35view0turn23search2

If Blurby’s question is instead, “Which model is most worth testing because it might genuinely improve narration quality over Kokoro?”, the answer is **OpenMOSS-Team**, with **Qwen** as the next track behind it. OpenMOSS is the more interesting Blurby prototype because it combines long-form ambition with explicit control surfaces that matter for readers, including duration control and emerging local deployment branches. Qwen may have the stronger raw long-speech evidence, but it is still the less attractive desktop-product fit. citeturn27view0turn27view3turn31view0turn39search0turn24view3turn40search1turn40search7

The most important bottom line is this: **no provider here clearly delivers a more natural long-form narration experience than Kokoro while also matching Kokoro’s local deployability and operational affordability inside Blurby’s current Windows-first architecture**. The right decision is therefore two-track. Keep Kokoro as the shipping lane. Prototype OpenMOSS as the only serious upgrade candidate. Keep Qwen and Chatterbox on the watchlist.
